# Ravana — Implementation Task List

Detailed task breakdown for each phase in [ARCHITECTURE.md §12](ARCHITECTURE.md). §12 is the *why this order* (architectural sequencing); this file is the *what to actually build*. Section references (§x.y) point back to ARCHITECTURE.md.

Phase 0 is broken down to the task level since it's what's actionable now — and further split into **0a/0b**, because "MVP" originally bundled a full vertical slice (schema, structured-output enforcement, LLM fallback, MCP, Docker sandbox, git isolation, HITL, audit log, mock mode, E2E) into one 4–6 week estimate, which isn't an MVP-sized unit of work, it's most of the platform. Splitting it separates the part that's genuinely hard and novel (the graph engine's routing/HITL/concurrency logic — already the source of two real bugs found just from writing docs and examples) from the part that's comparatively mechanical integration work (real LLM calls, real tool execution, real git). Phases 1–3 are intentionally less granular — per §12's own reasoning, each phase should surface what the next one actually needs, so committing to a detailed Phase 2 task list before Phase 0 ships would mostly be guessing.

(3) Transient-vs-hard at chain exhaustion is judged on each entry's **terminal outcome**, not any historical failure — an entry whose 500 retry died on a 401 ended permanent, so an all-permanent-ending chain raises hard instead of letting one early 500 buy `max_retries_per_node` node retries. ### Toolkits
- [x] **Toolkit executor interface, with §8(a) JSON-schema in / string out**: `ToolkitHandler` declares an `input_schema` (each connector's typed input contract, per §8(a)); the result is a string fed back to the model, so there's no separate output schema. `RavanaToolExecutor` implements the gateway's `ToolExecutor` seam, resolves the tool name to a handler, and dispatches.
- [x] **§8(a) input enforced in the runtime, not just declared**: `RavanaToolExecutor` validates arguments against `handler.input_schema` (shared `validate_json`, `runtime/schema_validate.py`) **before** dispatch and rejects a violation with a `ToolkitError` — a provider-side tool schema is a hint the model usually follows, not a safety boundary, so a malformed/injected call never reaches the connector and nothing is written to the ledger for it.
- [x] **Idempotency ledger scoped to side effects, method-aware** (`tool_invocation` table, §2.2): a *side-effecting* call's retry under the same content-addressed key returns the stored result instead of re-firing the side effect (§3.6/§8); a **read-only** call is not deduped (a repeated GET/poll returns live state, not a cached replay). A FAILED row is never reused. Whether a call is side-effecting is decided per-call by `ToolkitHandler.is_side_effecting(arguments)` — **not** a static flag — because one handler is read-only or mutating depending on args (an `api_connector` GET vs POST); the executor consults this method for both the pre-dispatch dedup lookup and the ledger write.
- [x] `api_connector` toolkit (generic HTTP): injectable httpx client, `config.base_url`, content-addressed key forwarded as `Idempotency-Key` header so the remote can dedupe too. Per **§8(c)** the connector does NOT resolve secrets — the registry (runtime) injects a dispatch-time credential *provider* (lazy — and **re-resolved on every call**, not memoized for the handler's lifetime, so a rotated token is picked up per dispatch like the LLM path; review finding) so the connector never holds the `auth_ref` or the resolver, and a run that never calls the toolkit never needs its secret; the provider returns `ResolvedSecret`, opened to plaintext only at the HTTP boundary. `EnvSecretResolver` resolves `auth_ref`/`api_key_ref` from `RAVANA_SECRET_*` (Vault backing is Phase 2, §8). **§8 hardening:** (1) a model-controlled `path` is rejected if it is an absolute or scheme-relative URL (would bypass httpx's `base_url` and send the bearer token off-base — token exfiltration via prompt injection) — validated **before** the token is attached, so a rejected request never carries the credential; (2) the HTTP method is validated against an allow-list; (3) `is_side_effecting` is method-aware (GET/HEAD/OPTIONS read-only, everything else mutating). `httpx` is now a direct dependency (the connector owns it) rather than a transitive one.
- [x] 18 test cases (15 tests, one parametrized ×4), all against fakes/no network: method-aware side-effect dedup (POST fires once on retry, GET/read-only NOT deduped), failed-then-retried, unknown tool, executor rejects args violating `input_schema` (never dispatched, ledger untouched), request shaping + injected auth + idempotency header, HTTP error, off-base absolute/scheme-relative/no-leading-slash path rejected with the token never sent, base-relative path allowed, disallowed method rejected, GET-vs-POST side-effect classification, deferred-type errors, `input_schema` declared, secret missing/scheme errors.

- [ ] `web_search` (provider HTTP, e.g. Tavily) — deferred; it's an `api_connector`-shaped handler against a search provider, lands with the connector-providers work.
- [x] **Surface an agent's real toolkits as callable tools to the model** (§3.4.4): the gateway's tool loop now offers `[*agent toolkits, submit_result]`, not `submit_result` alone. `ToolkitHandler` gained a `description` (author-provided descriptions in the manifest are a follow-up; each handler supplies a default) and the `ToolExecutor` protocol gained `tools_for(toolkit_ids) -> list[Tool]` (RavanaToolExecutor builds specs from the handlers, `_NoToolExecutor` returns none so a run with no wired toolkits stays `submit_result`-only). Two correctness points fell out: (a) the one-shot **guided** path can't express tool calls, so an agent *with* toolkits runs the tool loop even on a guided-capable provider (otherwise it would silently never see its tools); (b) the tool loop now has an **absolute ceiling on model round-trips** so a non-compliant provider that ignores the forced `submit_result` fails the turn instead of looping forever. A toolkit id colliding with the reserved `submit_result` is rejected (a compile-time reserved-id check is a cleaner future home).
- [x] **Per-node tool boundary enforced at execution, not just in the prompt** (§8 / ARCHITECTURE §916): the offered-tools list is a prompt-level hint; the gateway now also *refuses to execute* any tool the agent wasn't granted (a prompt-injected call to another registered tool gets an error tool_result, never dispatched). The `max_tool_calls_per_turn` guard is enforced **within a single multi-call response** too, so a batch of tool calls can't fire side effects past the budget before the next force-submit — each surplus call gets a "budget exhausted" error result instead of executing.
- [x] **Deferred toolkits are not advertised as callable tools**: a non-executable handler (`code_interpreter`/`mcp_server`/`web_search`, the sandbox slice) carries `executable = False`, and `tools_for` **raises** rather than offering it — surfacing it would only invite the model to call a tool guaranteed to fail. Consequence (honest): a real `--backend llm` run requires *every* toolkit an agent declares to be executable, so **the SDLC example workflow can't run under `--backend llm` until the sandbox slice lands** (its `dev`/`qa`/`sa` agents use `code_interpreter`/`mcp_server`); a workflow using only `api_connector` runs today.
- [x] **Turn-failure boundary hardened** (review round 2): (1) the engine catches *any* non-transient error from `run_turn` — a deferred/unknown toolkit surfaced by the gateway, a `submit_result`-id collision, an unexpected bug — and records a clean FAILED run + FAILED node_execution via `_fail_run`, instead of a process crash that stranded the node in `RUNNING`. Only `TransientAgentError` is still retried. (2) A toolkit *execution* failure (`ToolkitError` from `.execute`) is now fed back into the turn as a tool error the model can react to, not raised — a tool failing is a normal agent-loop event, not a run crash. (3) A response mixing `submit_result` with real tool calls no longer drops the tool calls: the real calls run and *each* premature `submit_result` (a provider can emit more than one) is deferred with an "ignored" notice — every `tool_use` in the recorded assistant turn gets a matching `tool_result`, so the transcript stays balanced (a strict provider like OpenAI rejects an unbalanced next request) and a payload can't claim work that never happened. (4) The gateway reports the **executed** tool-call count (not refused/over-budget attempts) so the engine's post-turn `max_tool_calls_per_turn` guard reflects real side effects and won't fail a run the gateway already capped.

### Definition of Done (found missing in the v0.16 design review — was parsed and persisted but never evaluated, and no task tracked it)
- [x] **DoD evaluator, gated at Terminate** (§3.1 step 7 — decided: a *check inside Terminate*, not a synthetic final node): a run that reaches a terminal `COMPLETED`s only if its `definition_of_done` is met, else `FAILED`s with the unmet criteria named; a `DOD_EVALUATED` event records the outcome either way (event type added to the §4 taxonomy). `engine/dod.py`'s `evaluate_dod` classifies each criterion by **parse** (`engine.expr.is_boolean_expression`), not by evaluation — a criterion that parses as an expression is enforced deterministically through the existing condition engine, and one that only *errors* at eval (e.g. `state.count > 5` on an unset key, where `None > 5` raises) is a **failing expression** (unmet, gating), NOT downgraded to advisory prose; anything that fails to parse (a sentence) is prose. Prose criteria go to a `prose_verdict(evaluated_by, criteria, state)` function that is **injectable at the engine boundary** (`start_run`/`resume_hitl` → `_RunCtx.dod_prose_verdict` → the Terminate gate), so prose is enforceable end-to-end, not just in unit tests; **when no verdict is wired, prose criteria are recorded as *unevaluated* (advisory — logged, not gating), not silently passed and not hard-failing every run.** The gate runs outside `_dispatch`'s failure boundary, so it has its own: a `dod_prose_verdict` that *raises* fails the run closed (FAILED, `ended_at` set) rather than escaping `start_run`/`resume_hitl` and stranding the run at RUNNING. 18 tests — 10 pure evaluator (expression pass/fail, missing-key-is-false, ordering-error-is-a-failing-expression, ordering-met, membership-error-is-a-failing-expression, prose advisory, prose enforced via a fake verdict, verdict receives only prose + `evaluated_by`, mixed classification, empty-is-vacuous) + 8 engine (unmet-expression → FAILED with the criterion recorded, met → COMPLETED, prose-doesn't-block-but-is-recorded, no-DoD → COMPLETED with no event, erroring-expression → FAILED, injected verdict gates COMPLETED/FAILED, verdict receives `evaluated_by` + final state, throwing verdict → FAILED not stranded).
- [ ] **Agent-backed prose DoD evaluator**: the `prose_verdict` seam is now wired through to `start_run`/`resume_hitl` and tested end-to-end with a fake verdict — the remaining half is a *default* verdict function that runs a real `evaluated_by` agent turn (via the LLM Gateway) judging the natural-language criteria against final state and returning a per-criterion pass/fail, plus CLI plumbing to install it. Until that default is wired, prose criteria are advisory unless a caller passes `dod_prose_verdict`.
- [x] **Real exponential backoff on the engine's per-node transient retry** (§3.6): a `TransientAgentError` now waits `backoff_delay(streak)` (shared `runtime/backoff.py`; equal jitter — half the exponential fixed + half uniform-random, keeping a deterministic `exp/2` floor tests can assert; base 1s doubling per **consecutive** failure, cap 30s) before re-queueing the node, still bounded by `guards.max_retries_per_node`. Keyed on the consecutive-failure streak (the same counter that bounds retries), **not** the node's lifetime `attempt` number — a node re-entered by a §3.7 loop or HITL resume has SUCCEEDED attempts that would otherwise inflate its first failure's delay toward the cap (review finding, regression-tested). And when the failure that just happened already exhausts the budget, the engine skips the backoff entirely — the re-queued dispatch fails the run without running a turn, so the final sleep bought nothing (review finding, regression-tested). Timing constants are module-level, not `guards` fields — §4's guards govern budgets, not timing; configurability waits for a workflow that needs it. The waiter threads through `start_run`/`resume_hitl` (`retry_sleep=`, default `asyncio.sleep`) so the suite records delays instead of sleeping (full suite still ~1s). 7 tests in test_backoff.py: delay doubling, cap, jitter bounds, 1-indexed guard, growing delays across consecutive failures, loop-revisit-doesn't-inflate-delay, no-sleep-before-a-guaranteed-budget-failure.
