# Ravana — Implementation Task List

Detailed task breakdown for each phase in [ARCHITECTURE.md §12](ARCHITECTURE.md). §12 is the *why this order* (architectural sequencing); this file is the *what to actually build*. Section references (§x.y) point back to ARCHITECTURE.md.

Phase 0 is broken down to the task level since it's what's actionable now — and further split into **0a/0b**, because "MVP" originally bundled a full vertical slice (schema, structured-output enforcement, LLM fallback, MCP, Docker sandbox, git isolation, HITL, audit log, mock mode, E2E) into one 4–6 week estimate, which isn't an MVP-sized unit of work, it's most of the platform. Splitting it separates the part that's genuinely hard and novel (the graph engine's routing/HITL/concurrency logic — already the source of two real bugs found just from writing docs and examples) from the part that's comparatively mechanical integration work (real LLM calls, real tool execution, real git). Phases 1–3 are intentionally less granular — per §12's own reasoning, each phase should surface what the next one actually needs, so committing to a detailed Phase 2 task list before Phase 0 ships would mostly be guessing.

## Phase 0b — Real Integration (LLM, tools, sandbox, git)

Goal: swap the mock backend for real providers and real tool execution, on a graph engine already proven correct in 0a — this phase is comparatively mechanical precisely because 0a already absorbed the hard, novel, bug-prone part.

Clients are injectable so tests never hit the network.

A transient failure retries with backoff while retaining the same logical visit and call ordinal, so a side effect reuses its invocation key; fatal failures fail the run, and model-addressable failures return an error tool result.

- [x] **Typed error taxonomy — retries only for genuinely transient failures** (§3.6, review finding: the old taxonomy caused wrong-type retries at several points): `ProviderError` carries `retryable`, and adapters classify every SDK failure via `classify_retryable` (429/5xx/408 → retryable; auth 401/403, invalid-request 400/422, unknown-model 404 → permanent; **no-status errors split**: builtin programming/config exceptions — TypeError/ValueError/KeyError/AttributeError, e.g. the SDK's missing-credential TypeError — are permanent, everything else without a status (connection reset, timeout, unknown SDK error) stays retryable as availability bias; the stale docstring — which claimed the gateway never retries ProviderError — rewritten to the real contract). The gateway acts on it: a **permanent** failure skips the same-entry retry *and its backoff sleep* and moves straight to the next fallback entry; a chain whose **every** entry failed permanently raises a hard error (not `TransientAgentError`), so the engine fails the run on the *first* node_execution instead of burning `max_retries_per_node` re-running a hopeless chain (engine-level test: 1 execution row, 0 sleeps). ### Toolkits
- [x] **Toolkit executor interface, with §8(a) JSON-schema in / string out**: `ToolkitHandler` declares an `input_schema` (each connector's typed input contract, per §8(a)); the result is a string fed back to the model, so there's no separate output schema. `RavanaToolExecutor` implements the gateway's `ToolExecutor` seam, resolves the tool name to a handler, and dispatches.
- [x] **§8(a) input enforced in the runtime, not just declared**: `RavanaToolExecutor` validates arguments against `handler.input_schema` (shared `validate_json`, `runtime/schema_validate.py`) **before** dispatch and rejects a violation with a `ToolkitError` — a provider-side tool schema is a hint the model usually follows, not a safety boundary, so a malformed/injected call never reaches the connector and nothing is written to the ledger for it.
- [x] **Idempotency ledger scoped to side effects, method-aware** (`tool_invocation` table, §2.2): a side-effecting invocation atomically commits a `STARTED` claim **before dispatch**, then records `SUCCEEDED`/`FAILED`; a retry under the same logical-invocation key returns the stored success instead of re-firing. An unresolved `STARTED` outcome left by process death or sandbox-cleanup failure fails closed rather than risking a duplicate side effect. Read-only calls bypass the ledger and an ordinary `FAILED` row may be retried. `ToolkitHandler.is_side_effecting(arguments)` decides per call because one handler may serve both GET and POST.
- [x] `api_connector` toolkit (generic HTTP): injectable HTTP client, `config.base_url`, and the logical-invocation key forwarded as `Idempotency-Key`. Credentials resolve lazily per dispatch and open to plaintext only at the HTTP seam. The connector rejects off-base URLs and disallowed methods before attaching auth, classifies failures by status/type, rejects credential echoes before they become model input, and closes clients it owns.
- [x] 18 test cases (15 tests, one parametrized ×4), all against fakes/no network: method-aware side-effect dedup (POST fires once on retry, GET/read-only NOT deduped), failed-then-retried, unknown tool, executor rejects args violating `input_schema` (never dispatched, ledger untouched), request shaping + injected auth + idempotency header, HTTP error, off-base absolute/scheme-relative/no-leading-slash path rejected with the token never sent, base-relative path allowed, disallowed method rejected, GET-vs-POST side-effect classification, deferred-type errors, `input_schema` declared, secret missing/scheme errors.
- [x] `code_interpreter` (local OCI runtime, Docker or Podman, §8/§10.1): the agent submits `code`; the handler atomically publishes it into `runs/<run_id>/workspace` under a forced bare filename, replacing rather than following a sandbox-created symlink, and rejects workspace/run-directory aliases. Execution sits behind an **injectable `SandboxRunner`** and Phase 0b accepts only the explicit local `docker`/`podman` backends rather than silently crossing trust boundaries. The pure `build_docker_argv` enforces `--rm`, **hard-disabled network egress** (`--network none`; any opt-in is rejected until a real per-host allow-list exists), hard memory/CPU/PID/per-file quotas, an aggregate workspace byte/file budget, read-only root + capped `/tmp`, dropped capabilities/no-new-privileges, the workspace owner's numeric UID/GID, disabled daemon-side log persistence, and exactly one run-scoped bind mount. `DockerSandboxRunner` drains attached stdout/stderr continuously while retaining only a bounded prefix and preserving the omitted-byte count; an unguessable launcher sentinel proves the in-container command actually started, so every terminal exit or timeout before that proof remains a retry-safe infrastructure failure rather than a successful agent result. Container provisioning completes `create` and captures its immutable ID before issuing `start --attach`, so a daemon-side create delayed past client cancellation can never start agent code. Process creation, execution, quota enforcement, timeout and cancellation are bounded: recursive quota scans run in killable/reaped worker processes rather than unbounded executor threads, while terminal output drain reserves part of one shared hard cleanup deadline for force-removal and daemon verification of the known ID. A terminal attach-client exit is accepted only after the daemon confirms that immutable ID is absent; otherwise Ravana force-removes it and fails closed. Per-workspace ownership uses cancellable, deadline-bounded acquisition and spans host-side capacity measurement, atomic publication and sandbox completion, so concurrent calls cannot overwrite an active invocation's script. A clean post-start timeout/non-zero agent exit is a normal result; failed cleanup or an otherwise unknowable post-start outcome is fatal and retains the `STARTED` ledger claim, while a sandbox that provably never started remains TRANSIENT. Regression tests cover argv isolation, traversal/symlink containment, strict backend/network config, limits/runtime mapping, bounded output, workspace budgets, startup proof on all exits/timeouts, create-before-start identity, delayed create/start clients, terminal daemon verification, repeated cancellation, one-deadline cleanup with reserved removal time, killable quota workers, monitor-shutdown cancellation, concurrent host publication, same-filename execution ownership, cancellable lock contention, workspace scoping and crash-safe dedup. Git-isolated provisioning is now owned by this handler through the runtime preparation boundary (next bullet), as is the terminal patch handoff that hands the workspace back.
- [x] `mcp_server` (stdio transport, official `mcp` Python SDK, §1.7): workflow config references an admin-owned `config.server` name; `.ravana/config.yaml` supplies the complete immutable command/args/env/cwd definition plus `auth_env`, optional authenticated discovery, and admin-owned `read_only_tools`, so workflow authors cannot alter process startup or side-effect classification. Tool lists use provider-compatible qualified names, are discovered only for servers granted to the active node, pinned per run, and durably snapshotted across HITL resume through a dedicated snapshot store; restored names are revalidated before advertisement, and pre-cwd snapshots fail closed because their former ambient execution directory was not persisted and cannot be verified safely. Timestamped orphan cleanup preserves snapshots belonging to concurrent preparation while eventually collecting crash debris. Runs are released independently from shared runtimes, each dispatch uses a fresh task-affine session and credential environment, credential-bearing child stderr is never forwarded directly to the host, every exposed tool field passes the secret-output gate, and a post-dispatch call or cleanup failure retains `STARTED` rather than becoming retryable `FAILED`. The logical idempotency key is sent as MCP metadata, pre-dispatch cancellation is retry-safe, and qualified-tool collisions are rejected before tools are advertised. `env.PATH` must contain absolute directories: a relative entry makes `which` return a relative path, and the child is spawned with the definition's `cwd`, so a relative command would resolve against a different directory than the parser validated and launch another binary with this server's credential environment. Normalising it was not enough — that only moved the ambiguity to parse time, where the same admin config resolved from one invocation directory and was rejected from another, so relative entries are refused outright and the resolved command is `abspath`ed as a second line. Its final symlink is deliberately **not** resolved: a virtualenv interpreter *is* a symlink to a base interpreter, and following it discards `sys.prefix`, so `.venv/bin/python3` became the bare CPython and the server could no longer import its own dependencies (the ordinary setup for a Python MCP server, reproduced). That regression survived a green suite because every launch test built a definition by hand while every parser test stopped at the parsed value, and one parser test asserted the rewritten path as expected — so a test encoded the defect. Because the launch path is the symlink, the definition fingerprint cannot take identity from it: it also records the resolved target — derived in the definition's own `__post_init__`, so identity belongs to the type rather than to the parser path — and repointing the link invalidates a restored tool snapshot instead of trusting it against a different binary. The parser→launch seam is now covered: a definition parsed from the venv interpreter is actually started and called, plus a **controlled** symlink test so the distinction holds on runners where `sys.executable` happens not to be one. **Not mitigated — §8 prompt injection**: a server-supplied tool *description* reaches the model verbatim, embedded by both adapters into the tool-definition surface it reads as authoritative framing, re-read every turn. Tool-list pinning defends the rug-pull (no post-approval swap) and the admin-curated endpoint is the reason to trust the server at all, but neither makes the text safe to read; the secret-output gate applied to it is leak prevention, not an injection boundary. The same gap is systemic, not MCP-specific: `api_connector`/`web_search` results and `code_interpreter` output also reach the model with no content-level envelope (both adapters do separate them into a `tool`/`tool_result` role, so the channel is distinct — the payload is not delimited), and `_format_result` frames stdout/stderr with plain-text labels the payload itself can forge. §8's "wrap/tag tool output distinctly" is unimplemented across all of them and wants one boundary, not four. HTTP/SSE remains deferred to the hosted tier.

## Cloud Product Requirements (Phases 1-3)

The hosted path must support the same workflow semantics as Local/Embedded while changing who owns execution and how it scales:

- [ ] **Multi-user / multi-workflow control plane**: authenticated users belong to an `org_id`; organizations can author and publish many versioned workflows, start concurrent runs, inspect history, and respond to HITL without sharing credentials or data across tenants. Every Run remains pinned to the workflow version it started with.
- [ ] **Default-isolated execution context** (§8.1): `org_id` is the hard tenant boundary and `run_id` is the runtime-context boundary. A Prompt is assembled only from that Run's state/transcript plus explicitly-authorized shared definitions or opt-in long-term memory. Same-user or same-org Runs never inherit one another's messages, state, HITL answers, artifacts, workspace or credentials automatically.
- [ ] **Two repository execution modes**: (a) fully cloud execution clones a repository with a short-lived Git provider installation token into an ephemeral sandbox; (b) hybrid execution dispatches signed jobs over an outbound-only connection to an organization-managed Local Runner for private/on-prem repositories. Neither mode executes agent code in the API process or against a user's live checkout.
- [ ] **Durable distributed execution**: stateless Orchestrators and workers claim leases, survive deploys/crashes, enforce per-org concurrency/rate limits, and keep HITL waits durable for hours or days.
- [ ] **Auditable usage metering**: append-only usage events record organization/user/run/node/provider/model, input/output/cached tokens, tool and sandbox compute, storage/egress, a pricing-version snapshot, and a unique source id for deduplication. `node_execution` token/cost fields remain operational rollups, not the billing source of truth.
- [ ] **Commercial billing and governance**: plans, quotas, prepaid credits or metered invoicing, budget alerts/hard caps, refunds/credits, and provider-invoice reconciliation. Platform-funded model usage and BYOK usage are priced separately because Ravana bears different costs.

Goal: `docker compose up` gives one authenticated team the Ravana Console with all three UI surfaces (§1.5), real concurrency, cloud-safe repository execution, and the usage data foundation required for later billing.

- [ ] **Data layer**: Postgres DDL from §2.2 (the real thing, not the SQLite translation); Alembic migrations from here on; one-time import path from a Phase-0 `.ravana/state.db` so early local runs aren't stranded
- [ ] **Orchestrator scaling**: `node_execution` lease claiming (`UPDATE ... WHERE status='QUEUED'` / `FOR UPDATE SKIP LOCKED`); `state_version` CAS commit with merge-policy-aware retry (§3.5) now actually exercised; broadcast-edge parallel dispatch; Redis Streams as the lease/HITL transport
- [ ] **Scheduler tick to unblock `queue`d concurrency runs** (§3.7 — 0a strands them `PENDING`; the Phase 1 scheduler owns dispatching a queued run when its group's active run reaches a terminal status). Also re-verify §3.8's join quiescence rule under multi-worker dispatch — "queue empty" needs a distributed equivalent (no leasable/leased node_executions for the run)
- [ ] **Maintain `run.current_nodes`** (§2.2 — reserved-but-unmaintained in 0a; the Operator view needs it)

## Phase 2 — Production hardening (scaled self-hosted)

Goal: multi-team, multi-domain org running Ravana on Kubernetes with real tenancy.

- [ ] Kubernetes Helm chart (API / orchestrator / worker pool / sandbox pool as independently-scaled deployments, per §10.3's diagram)
- [ ] `org_id` enforcement across every query and API auth check, backed by cross-tenant isolation tests (multi-tenancy stops being theoretical)
- [ ] RBAC roles (Workflow Author / Operator / Viewer) and auth middleware
- [ ] **Context-isolation enforcement** (§8.1): server-derived tenant identity; Postgres RLS; namespaced cache keys, Vault/KMS paths, object-storage prefixes, vector collections and usage events; one workspace/sandbox per Run; worker ownership revalidation before context load. Add negative tests for every cross-tenant API/resource path and concurrent-Run contamination tests for prompts, caches, artifacts, logs, metering and provider routing.
- [ ] Vault/KMS integration replacing any placeholder secret resolution
- [ ] Managed sandbox becomes the default backend (not opt-in) — multi-tenant workflows are now the norm, per §8
- [ ] **Hybrid Local Runner**: organization-scoped runner registration, outbound-only authenticated job stream, signed/leased jobs, heartbeat/revocation, capability labels, and result/artifact upload. A disconnected runner leaves work resumable/reclaimable rather than losing the Run.
- [ ] Golden-run regression suite + eval harness gating `publish` (§11)
- [ ] pgvector long-term memory (§1.3)
- [ ] Workflow composition / sub-workflows (§1.8): `workflow_node.sub_workflow_id`, nested-run dispatch, `output_map` commit — build only once a real cross-workflow reuse need shows up
- [ ] Per-org backpressure **and** per-credential rate limiting (§9) — the two are different failure modes, both needed
- [ ] **Usage governance and reconciliation**: per-org/project/user usage views; budget alerts and hard caps checked before dispatch; actual usage reconciled after provider response; pricing catalog distinguishes platform credentials from BYOK and retains historical versions.
- [ ] Loki + mandatory `run_id`/`node_execution_id` correlation across infra logs (§9)
- [ ] Stuck-run detection/alerting on `WAITING_HUMAN` timeout and heartbeat-less `RUNNING`

## Phase 3 — Scale-out / Managed Cloud

Goal: multi-region throughput, and a Managed Cloud tier that's an actual business, not just Phase 2's architecture with a login page.

- [ ] Kafka event bus, replacing Redis Streams for multi-region throughput
- [ ] Supervisor/dynamic LLM routing (§3.3) — an LLM-chosen `next_node` for topologies that don't reduce to boolean conditions
- [ ] Community Toolkit marketplace (submission/review process, registry, versioning)
- [ ] Data governance: enforce the `pii: true` state-key flag end-to-end, build a right-to-delete flow that redacts flagged content while preserving non-PII audit rows
- [ ] **Commercial billing**: aggregate immutable `usage_event`s by `org_id` and billing period; apply plan allowances/credits/markups/tax policy; generate customer-visible line items; synchronize subscriptions, invoices, payments, credits, and webhook reconciliation with a metered-billing provider (e.g. Stripe). Failed/retried work is charged or credited by an explicit product policy, never silently omitted.
- [ ] SOC2 compliance program (operational, not architectural — start the clock early since certification timelines are long)
- [ ] Multi-region deployment topology
